# ProChefDesk — Sürüm geçmişi

Mevcut sürüm: **v2.8.33**

Yapısal kilometre taşları (kronolojik tersine):

* **v2.8.33** — Invisible sync reliability (Faz 10 / Adım 2): Operatörün şu an gördüğü "her şey kendiliğinden çalışıyor" deneyimini sağlayan üç katmanlı koruma. (1) **Drift detection** `app/js/core/cloud.js`'in pull akışına eklendi. v2.8.3'teki "local-only workspace → cloud push" mantığı tüm 14 per-workspace tabloya (recipes, ingredients, menus, events, suppliers, canvases, shopping_lists, checklist_templates, stock_count_history, haccp_logs/units/readings/cook_cool) + inventory'e genişletildi: pull tamamlandığında local'de olup remote'ta olmayan her kayıt otomatik `queueUpsert` ile cloud'a iletiliyor. Restore başarısızlığı, transient push hatası, network kesintisi sonrası kalan local-only kayıtlar bir sonraki pull'da sessizce iyileştiriliyor — kullanıcı hiç farketmiyor. v2.8.32'deki manuel "force re-sync" hareketinin otomatik versiyonu. (2) **Auto-retry** `cloud-pertable.js`'e `_retryUpsert` ve `_retryDelete` helper'ları eklendi. Transient hatalar (no code = network, 5xx, 408, 429, timeout/network keyword match) için 1s + 2s backoff ile 3 deneme. Hard hatalar (RLS, 4xx schema mismatch) anında raporlanır — retry başarısız olacağı için zaman kaybetmez. (3) **Ambient sync indicator** `app.js` boot akışına `_installSyncIndicator()` eklendi. Viewport sağ alt köşede 10px floating dot. State'ler: syncing (mavi pulse animation), synced (yeşil, 2sn sonra otomatik solar/kaybolur), offline (gri, navigator.onLine=false algılandığında), error (kırmızı, tıklayınca queueFullState + flushNow retry). Event mekanizması: cloud-pertable.js flushNow başında `pcd-sync-status` DOM CustomEvent yayar (state: 'syncing'), tamamlandığında ('synced' veya 'error') yayar. Indicator pasif — default invisible, synced state'i 2sn yeşil tick gösterip kaybolur, sadece sorun anında kullanıcıya görsel bilgi verir. Şef "sync" kavramını bilmeden güven hissi alır. (4) **Force re-sync** butonu Account → Data & Sync'ten **Account → Yardım**'a taşındı (gündelik UI'da değil, troubleshooting aracı). i18n şef-dostu hale getirildi: "cloud push" → "saved to your account", "Force re-sync" → "Re-sync my data", sync_state_* tooltip'leri eklendi. 7 dosya değişti (cloud.js, cloud-pertable.js, account.js, app.js, en.js, tr.js, config.js).

* **v2.8.32** — Critical fix: silent cloud push failures + Force Re-sync button. Operatör backup restore yaptıktan sonra mobil cihazda sadece 2 recipe + 3 ingredient gördüğünü, masaüstünde 29 + 78 olmasına rağmen, raporladı. Root cause: `cloud-pertable.js`'in `flushNow` fonksiyonu her upsert/delete promise'in `.catch(e => PCD.warn(...))` ile yutuyordu — Supabase hata döndürse bile (RLS, network, schema, 5xx) chain devam ediyor, Promise.all başarılı resolve oluyordu. `account.js`'in restore handler'ı `await PCD.cloudPerTable.flushNow()` çağırıp başarılı sayıyordu, ardından 600ms sonra `window.location.reload()` tetikliyordu. Sonuç: cloud push tamamen başarısız olduğu durumda bile restore "başarılı" gözüküyor, reload sonrası diğer cihazlar boş cloud'dan pull yapıp boş kalıyordu — sessizce veri kaybı. Manuel kurtarma: DevTools console'da `PCD.store.listRecipes().forEach(r => PCD.store.upsertRecipe(r))` + `PCD.store.listIngredients().forEach(i => PCD.store.upsertIngredient(i))` + `PCD.cloudPerTable.flushNow()` ile 27 recipe + 75 ingredient cloud'a yeniden gönderildi, incognito tab anında güncellendi. **Fix:** (1) `flushNow` artık `{ success, pushed, errors[] }` döndürüyor — her upsert/delete hatası `errors` array'ine toplanıyor, eski warn-log davranışı korunuyor (backward compat). (2) `account.js` restore handler `flushNow` sonucunu kontrol ediyor: başarısızsa exception fırlatıyor, **reload YAPMIYOR**, kullanıcıya teknik mesaj gösteriyor ("Local restore succeeded but N cloud push operations failed: ... Click Force re-sync to cloud to retry"). (3) Account → Data & Sync'e "Force re-sync to cloud" butonu eklendi: confirm modal (recipe + ingredient sayısı görünür) → onay → `queueFullState()` + `flushNow()` → result toast (success/error). v2.8.33'te bu buton Yardım altına taşındı.

* **v2.8.5** — Realtime CHANNEL_ERROR fix (kozmetik, lansman polish). `app/js/core/cloud-realtime.js`'in `init()` fonksiyonunda iki subscribe path'i race ediyordu. (1) `PCD.store.on('user')` listener — auth `_setUser` → store change event → 1000ms `setTimeout(subscribe)`. (2) Fallback `setTimeout(subscribe, 2000)` — auth event hiç gelmezse diye konmuş savunmacı yedek. Akış sırası: SIGNED_IN gelir → store user set → 1. yol 1000ms sonra subscribe çağırır → channel henüz SUBSCRIBED'a geçmeden 2. yol 2000ms'de tekrar subscribe çağırır → ilk channel'ı `removeChannel()` ile cleanup eder → callback olarak CHANNEL_ERROR fırlatır. Sonuç: console'da kırmızı uyarı (`cloud-realtime: subscribe failed CHANNEL_ERROR`) ama 10sn retry akışı zaten kendiliğinden iyileştiriyordu — sadece kozmetik. **Fix:** `setTimeout(subscribe, 2000)` fallback path'i tamamen silindi. Gereksizdi: SIGNED_IN ve mevcut session getSession() iki yolu da `auth._setUser` → `store.set('user')` zincirini tetikliyor, fallback'in dolduracağı bir boşluk yok. Tek path kaldı, race yok. **Doğrulama:** Multi-device anlık sync testi — masaüstü Chrome'da recipe ekle → Android Chrome'da 1sn'den az gecikmeyle göründü, console temiz.
