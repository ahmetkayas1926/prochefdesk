# ProChefDesk — Sürüm geçmişi

Mevcut sürüm: **v2.7.1**

Yapısal kilometre taşları (kronolojik tersine):

* **v2.7.1** — Account silme akışından `user_data` DELETE çağrısı kaldırıldı. `js/tools/account.js`'in delete-account fonksiyonunda 3. adım olarak `supabase.from('user_data').delete()` çağrısı duruyordu — `user_data` tablosu v2.6.87'de DROP edilmişti, dolayısıyla bu çağrı her hesap silmesinde "relation does not exist" hatası fırlatıp `errors` array'ine sessizce ekleniyordu (kullanıcıya görünmüyordu çünkü `allOk: false` dönüyordu zaten). `deleteUserData` const tanımı, ilgili yorum ve `Promise.all` üçlüsünden referans silindi. Etki sıfır: silme akışı zaten photos + shares ile devam ediyordu, user_data hiçbir şey yapmıyordu.
* **v2.7.0** — Ölü kod temizliği: `cloud-migrate-v4` ekosistemi tamamen kaldırıldı. (1) `js/core/cloud-migrate-v4.js` (290 satır) silindi — Faz 4 Adım 3 blob → per-table migration script'i, görevini v2.6.73-92 sürecinde tamamladı, `migrationFazV4Done` flag'i tüm hesaplarda set olduğu için pratikte no-op çalışıyordu. (2) `js/core/auth.js`'ten 2 çağrı bloğu silindi: signed-in path (8 satır) ve existing-session path (7 satır promise chain adımı) — auth.js 302→287 satır. (3) `index.html`'den `<script src="js/core/cloud-migrate-v4.js">` satırı silindi (olmayan dosyaya 404 olmaması için zorunlu). Etki sıfır: dosya zaten boş dönüyordu. Boot süresi mikrosaniye seviyesinde kısalır, tarayıcı 1 daha az script indirir.
* **v2.6.99** — `listWorkspaces` soft-delete filter (yan etki kapatma). `js/core/store.js`'in `listWorkspaces` fonksiyonu sadece `archived` filter'ı yapıyordu, `_deletedAt` filter'ı yoktu. Ayrı bir SQL adımı olarak DB'ye eklenen workspace_tombstones cascade trigger (v2.6.98 DB-only) workspace silindiğinde `workspaces.deleted_at` ve 16 ws-scoped tablonun `deleted_at` kolonunu SET ediyor; `cloud-pertable.js`'in pull akışı bu değeri `state.workspaces[wsId]._deletedAt`'e yansıtıyor. Filter olmadığı için silinmiş workspace çok-cihazlı senaryoda veya Realtime UPDATE event akışında "isimsiz satır" olarak workspace selector'a sızabiliyordu. Tek satır fix: `Object.values(...)`'tan sonra `.filter(function (w) { return !w._deletedAt; })` zinciri eklendi. `includeArchived=true` çağrıları (app.js, account.js'te 3 yer) da bu filter'dan otomatik faydalanır — silinmiş workspace asla görünür alana çıkmaz. Trash UI ileride gelecek 3. bölümde (Active/Archived/Deleted) bu kayıtları ayrı bir endpoint üzerinden gösterecek.
* **v2.6.98** — Dead feature cleanup (Account ekranı sadeleştirme): Üç redundant özellik kaldırıldı. (1) **Photo storage cleanup UI** (v2.6.64'te eklenmişti): Account → Storage Tools bölümü altındaki manuel orphan foto temizleme aracı. v2.6.97'de server-side cleanup (Edge Function + pg_cron) gelince işlevsiz hale geldi — frontend cascade silme + server-side haftalık cleanup zaten her şeyi temizliyor, manuel UI her zaman "0 orphan" gösteriyordu. `openStorageAuditModal` (245 satır), tetikleyen buton, section başlığı, `runOrphanCleanup` ve dataURL migration mantığı silindi. (2) **Export Recipes (CSV)** ve **Export Ingredients (CSV)** butonları: Tarif CSV'si bütünlüğü kaybediyordu (steps, photo, plating, allergens yok) — gerçek bir migration veya cost analizi senaryosunda kullanılamıyordu (Cost Report PDF zaten daha iyi). Ingredient CSV'si tedarikçiye fiyat listesi gönderme senaryosu için tasarlanmıştı ama bu yön gerçek mutfak akışıyla uyuşmuyor (tedarikçi ChefDesk'e fiyat gönderir, tersi değil). Her iki buton, handler'lar ve associated toast mesajları silindi. (3) **Ölü i18n string'leri**: kullanılmayan `acc_export_*` key'leri (eski v17 fallback'ten kalan) temizlendi. Toplam silme: ~314 satır JS (account.js: 1754→1440), 34 satır i18n (en + tr). Diğer 4 dil (es/fr/de/ar) bu key'lere zaten sahip değildi (en fallback kullanıyorlardı). Frontend tarafında veri kaybı yok — JSON Download backup özelliği (data portability standardı) korunuyor. Server-side cleanup mekanizması (v2.6.97) etkilenmedi.
* **v2.6.97** — Faz 5 Adım 3: Server-side cleanup (DB + Storage). Önceki davranış: soft-deleted kayıtlar (Trash, 30 gün retention) lokal UI'da temizleniyordu ama DB'de sonsuza kadar kalıyordu — tablolar şişiyordu. Recipe foto'ları Storage'da orphan kalıyordu (recipe silinse bile dosya bucket'ta tutuluyordu). Bu sürümle iki yeni backend bileşeni: (1) `migrations/v2.6.97-cleanup-cron.sql` — `pcd_cleanup_old_deleted()` plpgsql function 17 ws-scoped tabloda + workspace_tombstones'tan `deleted_at < NOW() - 30 days` kayıtlarını fiziksel siler; pg_cron ile günlük UTC 03:00 schedule. (2) `supabase/functions/cleanup-photos/index.ts` — Edge Function service-role ile çalışır, recipes.data.photo URL referanslarını set'e koyar, recipe-photos bucket'ını listeler, set'te olmayan dosyaları batch DELETE eder; internal `x-cleanup-secret` header ile yetkilendirme. Frontend tarafında değişiklik yok — bu sürüm tamamen backend setup. Operatör deploy adımları: Dashboard'dan pg_cron extension enable, SQL migration çalıştır, Edge Function deploy (CLI), CLEANUP_SECRET secret'i ekle, Dashboard cron schedule (haftalık) opsiyonel.
* **v2.6.96** — Offline banner i18n + güven verici metin: `app.js`'teki offline banner metni hardcoded İngilizce idi (kullanıcı dili TR/ES/FR/DE/AR olsa bile İngilizce gösteriyordu) — `PCD.i18n.t('offline_msg', ...)` çağrısına çevrildi. Yeni metin tüm 6 dilde "değişiklikler güvende, otomatik senkronlanır" vurgusuyla yazıldı (operatör isteği: panik yerine güven). `offline_msg` çevirileri ar/de/es/fr dillerine eklendi (önceden sadece en/tr'de vardı). Banner'ın görünür/gizli mantığı dokunulmadı.
* **v2.6.95** — Faz 5 Adım 2: Cloud sync queue persistence. Önceki davranış: `cloud-pertable.queue` yalnızca JS hafızasında. Offline yazımda (Wi-Fi kesik mutfakta veya mobil bağlantı kötü iken) kullanıcı sekme kapatırsa kuyruk uçuyordu — cloud'a hiç gitmemiş yazımlar diğer cihazlarda görünmüyordu, hatta lokal IDB'de var olsa bile pull/sync döngüsünde çelişki yaratabiliyordu. Yeni davranış: her `queueUpsert`/`queueDelete` mutation'ı sonrası queue snapshot'ı 250ms debounce ile IDB'ye yazılıyor (key: `pertable_queue`). Boot sırasında `_loadPersistedQueue()` async tetikleniyor — kalan kuyruk varsa queue array'ine merge ediliyor, online + ready ise `scheduleFlush` çağrılıyor. `flushNow` Promise.all sonrası queue mevcut durumunu IDB'ye yansıtıyor (flush sırasında gelen yeni mutation'lar korunur). `wipeAllUserData` IDB queue'yu da temizliyor (Clear all sonrası phantom item'lar dönmesin diye). Race koruması: load sırasında canlı queue'da aynı `dedupeKey` varsa persisted item atlanıyor (yeni canlı kayıt daha güncel). Idempotent: upsert'ler `onConflict` ile, duplicate yazım güvenli. Etki: tarayıcı kapansa, cihaz uyusa, ağ kesilse — yazılan veri reload sonrası otomatik tekrar gönderilir.
* **v2.6.94** — Demo cleanup: Account ekranındaki "Re-add/Remove demo recipes" butonu tamamen kaldırıldı. Operatör kararı: kullanıcı için demo Add/Remove opsiyonu yok; misafir siteye girdiğinde tek seferlik seed olur (Fix #2 user-bağımlı gate sayesinde sign-in sonrası asla tetiklenmez), sign-in olunca `auth.js`'in `clearUserData` çağrısı tüm demo'yu temizler. Yapılan: (1) `account.js`'ten `demoToggleBtn` HTML ve click handler silindi; (2) `demo-recipes.js`'ten `removeDemo()` fonksiyonu kaldırıldı, public API artık `PCD.demo = { seed }`; (3) i18n string'leri (`clear_demo`, `reset_demo`, `demo_3_recipes`) 6 dil dosyasından (en, tr, es, fr, de, ar) silindi — Türkçe çeviriler dahil. Sıfır kalıntı referansı.
* **v2.6.93** — Faz 5 Adım 1: Misafir/üye demo ayrımı + welcome tour cloud-bağımsızlığı + backup restore tam zincir + Clear all cloud temizliği. **Dört fix tek pakette, birbirini destekliyor:** (1) `app.js`'in demo seed gate'i artık user-bağımlı (`if (!user && !demoSeeded)`) — sign-in olmuş kullanıcı için demo asla otomatik tetiklenmez; misafir ilk açılışta bir kez seed olur. Önceki davranışta her F5'te boot pull → cloud user_prefs.onboarding boş → demoSeeded=false → demo recipe duplicate ediliyordu (9→12→15 progression). (2) Welcome tour bayrağı state'ten localStorage'a taşındı (`pcd_tour_done`); `state.onboarding.mainTourDone` cloud-pertable sync'inde güvenilir yazılmıyordu, her F5'te tour geri açılıyordu. localStorage cloud-bağımsız, F5/sign-in/out'tan etkilenmez. State flag'i fallback olarak okunuyor (eski cihazlardan true gelen kullanıcılar için tour açılmasın). (3) Backup restore handler async/await zinciri: `wipeAllUserData` (RLS user_id ile 19 tabloda DELETE) → `replaceAll` (runMigrations + orphan ws cleanup + activeWorkspaceId validation + `flushSync` await) → `queueFullState` → `flushNow` await → reload. Önceki handler `store.set()` debounced + 800ms timeout idi, hiçbir adım garantili tamamlanmıyordu, cloud da temizlenmiyordu. (4) Clear all data handler artık `wipeAllUserData` çağırıyor; önceki davranış `cloud.pushNow()` idi ama bu v2.6.87'den beri no-op — Clear all aslında cloud'u hiç temizlemiyordu, reload sonrası pull her şeyi geri çekiyordu. **Yeni public API:** `PCD.cloudPerTable.wipeAllUserData()`, `PCD.cloudPerTable.queueFullState()`. `store.flushSync` ve `cloud-pertable.flushNow` artık Promise döndürür (eski sync caller'lar Promise'i discard ediyor, davranış aynı).
* **v2.6.92** — Faz 4 Adım 4c: localStorage yazma yolu kapatıldı + tek seferlik LS cleanup. `store.js`'e iki module-level flag eklendi (`_idbWriteOnly`, `_migrationDone`). İlk başarılı IDB persist'inden sonra `_completeMigration()` çağrılır: `state.prefs.idbWriteOnly = true` (kalıcı flag), `localStorage.removeItem(LS_KEY_STATE)`, modül flag'leri set. Boot'ta `load()` flag'i okur ve `_idbWriteOnly` aktif eder. `persist`, `flushSync`, `trimAndPersist` artık `_idbWriteOnly === true` iken LS yazımını atlar — sadece IDB'ye yazar. `reset()` sign-out döngüsünde flag'leri sıfırlar. Sonuç: localStorage 5 MB sınırı fiilen kaldırıldı; tek state kaynağı IDB. **Faz 4 tamamlandı.**
* **v2.6.91** — Faz 4 Adım 4b: Okuma yolu IndexedDB'ye geçti. `store.js` `load()` async oldu — önce IDB'den (`prochefdesk` DB, `state` store, `main` key) okuyor, IDB boşsa `localStorage` fallback'i. `store.init()` Promise döndürür. `app.js` `boot()` async function olarak hem `await PCD.store.init()` hem `await PCD.auth.init()` ediyor — auth zinciri (getSession → cloud.pull → photo migrate → cloud-migrate) tamamlanmadan `onAuthResolved()` çağrılmıyor, router register pull'dan SONRA, UI listener'lar pull-edilmiş state üzerine kuruluyor. Boot'un her adımına teşhis için `console.log('[boot] ...')` eklendi.
* **v2.6.89** — Faz 4 Adım 4a: IndexedDB altyapısı + write-through. Yeni dosya `js/core/idb-wrapper.js` (~110 satır, native IndexedDB üzerine Promise wrapper). `store.js`'in 4 yazma noktası (`persist`, `flushSync`, `trimAndPersist`, `reset`) artık LS'ye yazdıktan sonra IDB'ye de paralel yazıyor (fire-and-forget).
